* ¿Qué es Dape?

*Dape* (Debug Adapter Protocol for Emacs) es un cliente de depuración para Emacs que implementa el *Debug Adapter Protocol (DAP)*.

** ¿Qué es DAP?

El *Debug Adapter Protocol* es un protocolo estándar creado por Microsoft para la depuración de código. Es como LSP (Language Server Protocol) pero para debugging en lugar de edición de código.

*Ventaja principal*: Un único protocolo para depurar cualquier lenguaje. No necesitas aprender diferentes debuggers para cada lenguaje.

* Dape vs dap-mode

| Característica              | dape                          | dap-mode                      |
|-----------------------------+-------------------------------+-------------------------------|
| Mantenimiento               | Activo (2024)                 | Menos activo                  |
| Instalación de adaptadores  | Manual                        | Semi-automática               |
| Complejidad                 | Más simple, minimalista       | Más features, más complejo    |
| UI                          | Minimalista, REPL-based       | UI más rica (buffers, panels) |
| Dependencias                | Pocas                         | Muchas (lsp-mode, etc)        |
| Rendimiento                 | Más rápido                    | Más pesado                    |
| Filosofía                   | Unix-like: simple y explícito | Feature-rich                  |

** ¿Cuándo usar dape?

- Prefieres configuración explícita y simple
- Quieres menos dependencias
- Te gusta la filosofía minimalista
- Eres nuevo en debugging con Emacs (menos overwhelming)

** ¿Cuándo usar dap-mode?

- Necesitas UI rica con muchos paneles
- Quieres instalación más automática
- Ya usas lsp-mode y su ecosistema
- Necesitas features avanzadas específicas

* El Rollo de los Adaptadores

** ¿Qué es un Debug Adapter?

Un *debug adapter* es un programa que actúa como intermediario entre:

#+begin_src
  Emacs (dape) <--DAP--> Debug Adapter <--protocolo específico--> Debugger Real
#+end_src

** ¿Por qué existen?

Cada lenguaje tiene su propio debugger:
- PHP → Xdebug
- JavaScript → Chrome DevTools, Node debugger
- Python → debugpy
- Go → delve

El Debug Adapter *traduce* entre el protocolo DAP (estándar) y el protocolo específico del debugger.

** Ejemplo con PHP/Xdebug

#+begin_src
  Emacs + dape <--DAP--> vscode-php-debug <--DBGP--> Xdebug <--> PHP Runtime
#+end_src

1. *Xdebug* se ejecuta dentro de PHP y usa protocolo DBGP
2. *vscode-php-debug* traduce DBGP ↔ DAP
3. *dape* habla DAP con el adaptador
4. *Tu* interactúas con dape en Emacs

** ¿Por qué no directamente Emacs → Xdebug?

Porque cada debugger tiene su propio protocolo:
- Xdebug usa DBGP
- GDB usa MI protocol
- LLDB usa otro protocolo

Con DAP, solo necesitas *un cliente* (dape) que hable *un protocolo* (DAP).

* Instalación de Dape

** 1. Instalar dape en Doom Emacs

En =~/.doom.d/packages.el=:

#+begin_src elisp
(package! dape)
#+end_src

Luego: =doom sync=

** 2. Configuración básica

En =~/.doom.d/config.el=:

#+begin_src elisp
(use-package! dape
  :config
  ;; Aquí van tus configuraciones de adaptadores
  )
#+end_src

* Instalación de Adaptadores

** Proceso General

1. *Descargar* el adaptador (usualmente archivo =.vsix= o binario)
2. *Descomprimir* en directorio de adaptadores
3. *Configurar* dape para encontrarlo

** Directorio recomendado

#+begin_src bash
mkdir -p ~/.config/doom/debug-adapters
#+end_src

** Ejemplo: PHP/Xdebug (🎯 INSTALACIÓN AUTOMÁTICA)

*** Paso 1: Añadir función de instalación automática

En =~/.config/doom/config.el=:

#+begin_src elisp
;; Función para instalar vscode-php-debug automáticamente
(defun dape-ensure-php-debug-adapter ()
  "Instala vscode-php-debug adapter si no existe."
  (let* ((adapter-dir (expand-file-name "~/.config/doom/debug-adapters/php-debug"))
         (adapter-file (expand-file-name "extension/out/phpDebug.js" adapter-dir))
         (download-url "https://github.com/xdebug/vscode-php-debug/releases/download/v1.35.0/php-debug-1.35.0.vsix")
         (temp-file (make-temp-file "php-debug-" nil ".vsix")))

    (unless (file-exists-p adapter-file)
      (message "📦 Instalando vscode-php-debug adapter...")
      (make-directory (expand-file-name "~/.config/doom/debug-adapters") t)
      (url-copy-file download-url temp-file t)
      (message "✓ Descargado adaptador")
      (call-process "unzip" nil nil nil "-q" temp-file "-d" adapter-dir)
      (delete-file temp-file)
      (message "✓ Adaptador instalado en %s" adapter-dir)
      (if (file-exists-p adapter-file)
          (message "✅ vscode-php-debug adapter instalado correctamente")
        (error "❌ Error instalando el adaptador")))
    (file-exists-p adapter-file)))

;; Comando para reinstalar si hace falta
(defun dape-reinstall-php-debug-adapter ()
  "Reinstala vscode-php-debug adapter forzosamente."
  (interactive)
  (let ((adapter-dir (expand-file-name "~/.config/doom/debug-adapters/php-debug")))
    (when (file-exists-p adapter-dir)
      (delete-directory adapter-dir t)
      (message "🗑️  Adaptador anterior eliminado"))
    (dape-ensure-php-debug-adapter)))
#+end_src

*** Paso 2: Configurar dape con instalación automática

#+begin_src elisp
(use-package! dape
  :config
  ;; PHP con Xdebug 3.x - El adaptador se instala automáticamente
  (add-to-list 'dape-configs
               `(phpListen
                 modes (php-mode)
                 ensure dape-ensure-php-debug-adapter  ;; ← Instalación automática
                 command "node"
                 command-args (,(expand-file-name "~/.config/doom/debug-adapters/php-debug/extension/out/phpDebug.js"))
                 :type "php"
                 :request "launch"
                 :mode "listen"
                 :port 9003)))
#+end_src

*IMPORTANTE*: El adaptador se descarga e instala automáticamente la primera vez que ejecutas =M-x dape=.

*** Paso 3: Configurar Xdebug

En tu =php.ini=:

#+begin_src ini
zend_extension=xdebug.so
xdebug.mode=debug
xdebug.start_with_request=yes
xdebug.client_host=127.0.0.1
xdebug.client_port=9003
#+end_src

* Uso Básico de Dape

** Workflow típico

1. *Abrir archivo* PHP en Emacs
2. *Poner breakpoint*: =M-x dape-breakpoint-toggle= (o =SPC m d b= en Doom)
3. *Iniciar dape*: =M-x dape= → seleccionar =phpListen=
4. *Hacer request* a tu app PHP (curl, browser, etc)
5. Emacs se *pausará en el breakpoint*

** Comandos en el REPL de dape

Una vez pausado en un breakpoint:

| Comando    | Acción                               |
|------------+--------------------------------------|
| =next=     | Ejecutar siguiente línea (step over) |
| =step=     | Entrar en función (step into)        |
| =out=      | Salir de función (step out)          |
| =continue= | Continuar ejecución                  |
| =pause=    | Pausar ejecución                     |
| =restart=  | Reiniciar sesión de debug            |
| =quit=     | Terminar debug                       |

** Inspeccionar variables

En el REPL, puedes escribir:
- Nombre de variable: =$myVar=
- Expresiones: =$user->getName()=

** Atajos estilo IntelliJ (añadido 2026-05-19)

Para no cambiar el mapa mental viniendo de IntelliJ/PhpStorm, hay
atajos globales que reproducen las F-keys clásicas:

| Tecla        | Acción                  | Equivalente IntelliJ      |
|--------------+-------------------------+---------------------------|
| =F7=         | =dape-step-in=          | Step Into                 |
| =F8=         | =dape-next= (step over) | Step Over                 |
| =Shift+F8=   | =dape-step-out=         | Step Out                  |
| =F9=         | =dape-continue=         | Resume Program            |
| =Ctrl+F8=    | =dape-breakpoint-toggle=| Toggle Line Breakpoint    |
| =Ctrl+F2=    | =dape-quit=             | Stop                      |
| =Alt+F8=     | =dape-evaluate-expression= | Evaluate Expression    |

Las funciones =dape-*= dan =No active debug session= si no hay sesión
abierta, así que pulsarlas sin querer NO rompe nada.

* Personalización visual

** Línea de ejecución: magenta Spacemacs

Por defecto dape usa la cara =dape-source-line-face= sin definir
(=unspecified=) → hereda del tema y resulta casi invisible. La
customización añade un magenta sólido al estilo Spacemacs:

#+begin_src elisp
(custom-set-faces!
  '(dape-source-line-face
    :background "#bc6ec5" :foreground "#ffffff"
    :weight bold :extend t))
#+end_src

=:extend t= hace que el fondo se extienda a todo el ancho de la línea,
no solo al texto. Tonos alternativos:

| Color       | Sensación               |
|-------------+-------------------------|
| =#bc6ec5=   | magenta Spacemacs       |
| =#d33682=   | magenta Solarized       |
| =#c678dd=   | púrpura Doom One        |
| =#ff5fff=   | magenta neón (muy duro) |

Para cambiarlo en caliente sin recargar:
=M-x set-face-attribute= → =dape-source-line-face= → atributos.

* El flow completo: qué pasa cuando pulsas F8

La cadena de procesos y mensajes:

#+begin_src
F8 (key-binding)
  → dape-next (función elisp)
  → dape envía DAP "next" al adapter por stdin/stdout
       {"command": "next", "arguments": {"threadId": 1}}
       ↓
  → adapter (phpDebug.js) recibe DAP
  → adapter traduce a DBGp "step_over"
       step_over -i 42
       ↓
  → xdebug.so (cargado dentro del proceso PHP) recibe DBGp por TCP
  → Xdebug ejecuta UNA línea y se para
  → Xdebug envía DBGp "response" al adapter
       <response status="break" line="123" filename="..."/>
       ↓
  → adapter traduce a DAP "stopped event"
       {"event": "stopped", "reason": "step", "threadId": 1}
       ↓
  → dape recibe el event
  → dape mueve el cursor a línea 123 del fichero
  → dape pinta el overlay magenta
  → dape refresca el buffer *dape-info* con variables/stack/threads
#+end_src

Todo en menos de 50ms. Esa fluidez es DAP.

** El árbol de procesos durante un debug

#+begin_src
emacs --daemon (PID padre)
 │
 ├── node phpDebug.js  ← spawneado por dape al pulsar M-x dape
 │   └── escucha en 127.0.0.1:9003 (TCP) hablando DBGp
 │
 └── (otra ventana del usuario, terminal/vterm)
      direnv → nix shell → php bin/console foo
                            └── xdebug.so cargado
                                  → conecta a :9003
                                  → habla DBGp con phpDebug.js
#+end_src

*Punto clave*: dape NO lanza el código PHP. Tú lo lanzas en tu
terminal. dape solo escucha. Modelo "passive listen": el debugger
viene a mí. IntelliJ es lo mismo por debajo, pero te lo unifica.

* Workflow real con direnv (Vocento)

Asumiendo proyecto tipo =~/src/vocento/php-service.user-identity= con
=.envrc= = =use flake= y el =flake.nix= configurando =xdebug.mode=debug=
y =xdebug.start_with_request=yes=.

** Paso a paso

1. *Tener IntelliJ con el listener APAGADO*. Solo un proceso puede usar
   el puerto 9003. Si IntelliJ tiene activado "Listen for PHP Debug
   Connections" (el iconito del bicho en la esquina), apágalo.

2. En Emacs: =SPC p p= → eliges el proyecto → te lleva al directorio.

3. Abrir el fichero PHP donde quieres parar (controller, command, etc).
   Pon el cursor en la línea exacta y pulsa =Ctrl+F8= o
   =M-x dape-breakpoint-toggle=. Verás un círculo en el margen.

4. =M-x dape= → te pide config → escribe =phpListen= → Enter. Emacs
   queda escuchando en 9003. Verás un buffer =*dape-info*= o similar.

5. En tu terminal (con el direnv ya activo, ergo =php= y =xdebug=
   cargados), ejecuta lo que dispare el código:

   - CLI: =php bin/console alguna:cosa=
   - HTTP: levanta el server y haz =curl http://...= o request desde
     navegador.
   - Tests: =vendor/bin/phpunit --filter ...=

6. Xdebug conecta automáticamente a 9003 (gracias a
   =start_with_request=yes=). Emacs salta al breakpoint con la línea
   marcada en magenta.

7. F7/F8/F9/Shift+F8 para navegar. Alt+F8 para evaluar expresiones.

* Diagnóstico desde Emacs vivo

Si algo no funciona, NO recargues Emacs entero. Usa =emacsclient --eval=
desde shell para inspeccionar el daemon vivo:

** ¿dape está cargado?

#+begin_src bash
emacsclient --eval "(fboundp 'dape)"
# Debe devolver: t
#+end_src

** ¿La config phpListen sigue ahí?

#+begin_src bash
emacsclient --eval "(assq 'phpListen dape-configs)"
# Debe devolver la tupla con :port 9003 y la ruta del adapter.
#+end_src

** ¿El adapter existe en disco?

#+begin_src bash
ls -la ~/.config/doom/debug-adapters/php-debug/extension/out/phpDebug.js
#+end_src

** ¿Hay alguien escuchando en 9003?

#+begin_src bash
ss -lntp | grep 9003
# Si sale algo con node = phpDebug.js corriendo (dape activo).
# Si sale algo con java = IntelliJ acaparando. Apágalo.
# Si no sale nada y dape está supuestamente activo = dape mal arrancado.
#+end_src

** ¿Xdebug está conectando?

#+begin_src bash
# El flake configuró log en /tmp/xdebug.log con log_level=7 (verboso).
tail -f /tmp/xdebug.log

# Lineas tipicas:
#   [Step Debug] INFO: Connecting to configured address/port: 127.0.0.1:9003
#   [Step Debug] INFO: Connected to debugging client: 127.0.0.1:9003
#+end_src

Si ves "Could not connect" repetidamente: nadie escucha en 9003, o
firewall, o puerto mal configurado.

* Conflicto frecuente: puerto 9003 ocupado

El puerto 9003 lo escucha *uno y solo uno* en cada momento:

| Quien                | Cuando                                |
|----------------------+---------------------------------------|
| IntelliJ / PhpStorm  | con "Listen for PHP Connections" ON   |
| VSCode + PHP Debug   | con sesión de debug PHP arrancada     |
| dape (Emacs)         | tras =M-x dape= con =phpListen=       |
| nada                 | el resto del tiempo                   |

Si lanzas =dape= y el puerto está ocupado, dape falla silencioso o con
"address already in use". Soluciones:

- Apagar el listener del otro IDE.
- O cambiar el puerto en =flake.nix= (=xdebug.client_port=) y en la
  config dape (=:port=), pero sincronizar ambos lados.

* Instalación Automática de Adaptadores (RECOMENDADO)

** ¿Por qué automatizar?

La instalación manual de adaptadores es tediosa y propensa a errores. Con la función automática:

- ✅ Se descarga e instala automáticamente al usar dape por primera vez
- ✅ Verifica que el adaptador existe antes de usarlo
- ✅ Fácil reinstalación si algo falla
- ✅ Cero configuración manual

** Cómo funciona

La función =dape-ensure-php-debug-adapter= se ejecuta cuando:
1. Intentas iniciar dape (=M-x dape=)
2. El parámetro =ensure= en la config llama a la función
3. La función verifica si existe el adaptador
4. Si no existe, lo descarga e instala automáticamente

** Comandos útiles

- =M-x dape-reinstall-php-debug-adapter= - Reinstalar adaptador forzosamente
- Útil si la instalación falló o hay una nueva versión

** Estructura de directorios

Después de la instalación automática:

#+begin_src
~/.config/doom/debug-adapters/
└── php-debug/
    ├── extension/
    │   └── out/
    │       └── phpDebug.js  ← El adaptador
    └── ...
#+end_src

* Troubleshooting

** "Adapter connection shutdown"

Posibles causas:
1. El adaptador no está instalado correctamente
2. Node.js no está disponible
3. Ruta incorrecta al adaptador
4. El adaptador usa TCP en lugar de stdio

Solución:
#+begin_src bash
# Verificar que node existe
which node

# Verificar que el adaptador existe
ls ~/.config/doom/debug-adapters/php-debug/extension/out/phpDebug.js

# Si no existe, reinstalar:
# M-x dape-reinstall-php-debug-adapter

# Verificar que Xdebug está cargado
php -v | grep -i xdebug
#+end_src

*IMPORTANTE*: Asegúrate de que la configuración NO use =--server=9003= en =command-args=. El adaptador debe usar *stdio*, no modo servidor TCP.

** Xdebug no se conecta

#+begin_src bash
# Ver logs de Xdebug
tail -f /tmp/xdebug.log

# Verificar que el puerto está abierto
netstat -tuln | grep 9003
#+end_src

** Breakpoints no funcionan

1. Asegúrate de que =xdebug.start_with_request=yes= en php.ini
2. Verifica que el puerto coincide (9003)
3. Comprueba path mappings si usas Docker

* Ventajas de Dape sobre Depuración Tradicional

** Sin dape (print debugging)

#+begin_src php
<?php
echo "DEBUG: user = ";
var_dump($user);  // ← Esto
die();            // ← Y esto
#+end_src

Problemas:
- Ensuciar código con prints
- Reiniciar servidor cada vez
- Ver solo lo que imprimiste
- Olvidar quitar los prints

** Con dape

- Breakpoints temporales (no tocar código)
- Inspeccionar *todas* las variables
- Step through código línea por línea
- Ver call stack completo
- Evaluar expresiones on-the-fly

* Filosofía de Dape

Dape sigue la filosofía Unix:

#+begin_quote
"Do one thing and do it well"
#+end_quote

- *Minimalista*: Solo lo esencial para debugging
- *Explícito*: Tú controlas todo, nada mágico
- *Composable*: Se integra bien con otras herramientas de Emacs
- *Extensible*: Fácil añadir nuevos lenguajes

No intenta ser un IDE completo, solo un excelente debugger.
